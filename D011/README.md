# AI 데이터 인텔리전스 부트캠프 과제: 에펨코리아 '포텐 터짐(화제)' 게시판 분석 파이프라인

## 1. 개요

본 프로젝트는 동적 웹 페이지인 에펨코리아(fmkorea)의 '포텐 터짐 - 화제순(`best2`)' 게시판에서
실시간 인기글 목록을 자동으로 수집하고, 정제 및 시각화를 통해 "카테고리에 따라 화제성(추천수)과
참여도(댓글수)가 어떻게 달라지는지, 요일에 따라 화제글 발생 빈도가 다른지"를 분석하는 것을
목표로 합니다. 데이터 수집, 정제, 시각화까지 전 과정을 하나의 Jupyter Notebook에서 재현할 수
있도록 구현했습니다.

## 2. 크롤링 과정 (Data Collection)

`best2`는 서버 렌더링 HTML이라 원래는 `requests` + `BeautifulSoup`만으로 충분한 구조지만,
fmkorea가 **DDoS 방어 시스템(WASM 챌린지)**을 도입한 이후로는 JS를 실행하지 않는 요청은 실제
게시글 대신 HTTP 430 차단 페이지만 돌려받습니다. 이 챌린지 로직을 역산해서 우회하는 대신,
**Playwright**로 실제 headless 브라우저를 띄워 사이트가 의도한 대로 챌린지를 정상적으로
통과시키는 방식을 택했습니다.

* **정상적인 챌린지 통과**:
  * `page.goto(url, wait_until='domcontentloaded')`로 이동한 뒤, 챌린지 스크립트가 검증
    쿠키를 세팅하고 자동 새로고침을 마칠 시간(`RENDER_WAIT`)을 추가로 기다립니다.
  * `wait_until='load'`(모든 하위 리소스 대기)는 이미지가 많은 게시글에서 타임아웃이
    자주 발생해서, HTML 파싱이 끝나는 시점까지만 기다리는 `domcontentloaded`로 대체했습니다.
* **페이징 처리 및 상태 추적**:
  * 목록 페이지(`/best2?page=N`)를 1페이지부터 순서대로 방문하면서 게시글을 수집합니다.
  * 단순히 정해진 페이지 수만큼 도는 것이 아니라, **연속 빈 페이지 감지**
    (`MAX_CONSECUTIVE_EMPTY`) 로직을 적용했습니다. `best2`는 캘린더 날짜 조회가 아닌
    실시간 랭킹 피드라서 페이지가 깊어지면 결국 실제 게시글이 끝나는데, 연속으로 지정한
    횟수만큼 빈 페이지가 나오면 "실제 데이터가 끝났다"고 판단하고 그 자리에서 자동
    중단해, 존재하지 않는 데이터를 억지로 긁으려는 무의미한 요청을 방지합니다.
* **환경 이슈 해결(Windows + Jupyter)**:
  * Jupyter 커널이 pyzmq 호환을 위해 asyncio 이벤트 루프 정책을 Windows에서 `Selector`
    방식으로 바꿔두는데, Playwright(sync API)는 브라우저를 서브프로세스로 띄우기 때문에
    `Selector` 루프에서는 구동이 불가능합니다. 이를 별도 스레드(`ThreadPoolExecutor`) 안에서
    `Proactor` 정책으로 전환해 실행하는 방식으로 해결했습니다.
* **서버 부담 최소화**: 요청 사이에 `time.sleep(REQUEST_DELAY)`으로 딜레이를 두고, `robots.txt`에서
  일반 봇(`User-agent: *`)에게 명시적으로 허용된 경로만 수집합니다.

## 3. 데이터 처리 과정 (Data Processing)

* **타겟 데이터 추출**: 목록의 각 게시글(`li.li`)에서 아래 선택자로 필드를 추출했습니다.

  | 항목 | 선택자 |
  | :--- | :--- |
  | 제목 | `h3.title`의 `data-original-title` 속성 |
  | 카테고리 | `span.category a` |
  | 작성자 | `span.author` |
  | 추천수 | `a.pc_voted_count .count` |
  | 댓글수 | `.comment_count` |
  | 작성일자(원본) | `span.regdate` (`"N분 전"` / `"N시간 전"` / `"YYYY.MM.DD"` 세 형태) |

* **결측치 처리**: 카테고리가 빈 문자열인 게시글은 제거하지 않고 `'미분류'`라는 별도 그룹으로
  남겨 두었습니다.
* **타입 변환**: 문자열로 수집된 추천수·댓글수(`"1,234"`, `"1.2만"` 등)를 정수로 변환했습니다.
* **날짜 정규화**: 상대 시간 표기(`"N분 전"`/`"N시간 전"`)는 수집 시작 시각을 기준으로 역산하고,
  절대 날짜 표기(`"YYYY.MM.DD"`)는 그대로 파싱해서 하나의 `datetime` 컬럼으로 통일했습니다.
  이를 바탕으로 `작성일`, `요일`, `대분류`(카테고리 상위 항목) 파생 컬럼을 만들었습니다.
* **중복 제거**: 수집 도중에도 사이트가 계속 갱신되기 때문에, 게시글 URL을 기준으로
  `drop_duplicates`를 적용해 중복 수집분을 제거했습니다.
* **이상치 처리**: 추천수·댓글수의 극단값은 오류가 아니라 "실제로 크게 화제가 된 게시글"이므로
  제거하지 않고 그대로 분석에 포함했습니다(`best2` 자체가 이미 화제글만 모아 놓은 게시판이라
  분포가 원래도 위로 치우쳐 있는 것이 자연스럽기 때문).

## 4. 데이터 분석 및 활용 (Output & Visualization)

* **분석 질문**: 화제글의 카테고리에 따라 추천수(화제성)와 댓글수(참여도)가 어떻게 달라질까?
  그리고 요일에 따라 화제글 발생 빈도가 다를까?
* **시각화 4종** (게시글 수 상위 8개 카테고리 기준):
  1. 카테고리별 추천수 분포 — 박스플롯
  2. 카테고리별 댓글수 분포 — 박스플롯
  3. 추천수-댓글수 산점도 + 피어슨 상관계수
  4. 요일별 화제글 수 — 카운트플롯
* **데이터 영속성**: 정제 전 원본은 `fmkorea_best2_raw.csv`, 정제된 데이터는
  `fmkorea_best2_clean.csv`로 저장해서 정제 전후 비교가 가능하도록 했습니다. 그래프 4종은
  `chart1_category_recommend.png` ~ `chart4_weekday_count.png`로도 함께 저장됩니다.

## 5. 사용 방법

1. **Python 설치**: Python 3.10 이상이 설치되어 있어야 합니다.
2. **패키지 설치**: 터미널에서 아래 명령어를 입력하여 필요한 라이브러리를 설치합니다.

   ```
   pip install playwright pandas beautifulsoup4 matplotlib seaborn
   playwright install chromium
   ```

3. **노트북 실행**: `fmkorea_best2_scraper.ipynb`를 Jupyter에서 열어 위에서부터 순서대로
   (또는 Run All) 실행합니다. 로그인이나 별도 브라우저 조작 없이 headless로 자동 수집됩니다.
4. **결과 확인**:
   * 실행 중 진행 상황(몇 페이지째 수집 중인지, 몇 건 모였는지)이 셀 출력으로 표시됩니다.
   * 최종적으로 `fmkorea_best2_raw.csv`, `fmkorea_best2_clean.csv`, `chart1~4_*.png`가
     같은 폴더에 생성됩니다.

## 주요 특징

* **탐지 우회가 아닌 정상 통과**: 사이트의 DDoS 방어(WASM 챌린지) 로직을 역산해서 우회하지
  않고, 실제 브라우저로 정상적으로 통과시키는 방식만 사용합니다.
* **실제로 존재하는 데이터만 수집**: 연속 빈 페이지를 감지하면 자동으로 수집을 멈춰서, 존재하지
  않는 기간의 데이터를 억지로 채우거나 서버에 불필요한 요청을 반복하지 않습니다.
* **서버 친화적 수집**: 요청 간 딜레이, `robots.txt` 준수, 조회수처럼 꼭 필요하지 않은 항목은
  상세 페이지 방문 없이 수집 범위에서 제외해 요청 수를 최소화했습니다.
* **재현 가능성**: 노트북 하나(`fmkorea_best2_scraper.ipynb`)만 실행하면 수집부터 시각화까지
  전 과정이 재현되며, 원본과 정제본을 모두 저장해 정제 과정에서 무엇이 바뀌었는지 비교할 수
  있습니다.
